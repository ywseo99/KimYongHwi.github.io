# 1. Introduction
오늘날 대부분의 client는 API를 통해 필요한 데이터를 전달 받는데 이때 API는 client에서 필요로 하는 데이터에 대한 interface를 제공해야할 책임이 있다.

기존의 REST API는 데이터의 대한 interface가 추가될 때마다 새로운 endponit를 필요로 했고 API의 버전이 올라갈 수록 또는 API의 규모가 커질 수록 이런 endpoint는 무수히 늘어나 관리가 어려워진다.

이러한 문제점을 해결하기 위해 Facebook에서는 GraphQL이라는 새로운 API 표준을 개발하고 오픈소스화 했다.

GraphQL의 핵심은 client가 필요한 데이터를 정확하게 지정하여 API로부터 원하는 데이터만 전달 받을 수 있다. 또한 고정 데이터 구조를 리턴하는 여러 엔드 포인트 대신 GraphQL 서버는 단일 엔드 포인트 만 노출하고 클라이언트가 요청한 데이터로 정확하게 응답한다.

# 2. 장점
- Ask for what you need, get exactly that
  - client에서는 query를 통해 원하는 데이터를 받아올 수 있어 over fetching, under fetching이 없다.

- Get many resources in a single request
  - 예를들어 user 정보, user가 작성한 posts, user의 followers를 API로부터 받아온다고 가정해보자.
  - Rest
    - REST방식은 3개의 endpoint에 3번의 request를 보낸다.

# 3. Core concept

## 3.1 The Schema Definition Language (SDL)
GraphQL은 type system을 가지고 있으며, 이를 사용해 API의 스키마를 정의한다. 스키마를 작성하기 위해 사용하는 문법을 SDL이라고 한다.

SDL을 통해 다음을 정의할 수 있다.
- type
- Query
- Mutation
- Subscription

```
type User {
  id: ID
  name: String
  age: Int
  posts: [Post]
}

type Post {
  title: String
  author: Person
}

type Query {
  getUsers(from: Int, size: Int): [User]
}

type Mutation {
  createUser(name: String, age: Int): User
}

type Subscription {
  newUser: User
}
```

위 예시는 User와 Post에 대한 type을 정의하고 그에 대한 Query, Mutation, Subscription이다. 

User와 Post는 User를 기준으로 one-to-many 관계를 가지고 있기 때문에 User의 post field는 Array인것을 알 수 있다.

모든 GraphQL 서비스는 해당 서비스에서 쿼리 가능한 데이터들을 완벽하게 설명하는 타입들을 정의하고, 쿼리가 들어오면 해당 스키마에 대해 유효성이 검사된 후 실행된다.
type에 정의되지 않은 필드를 요구하거나 query(GET), mutation(POST, PUT, DELETE)를 실행할 때 또는 Query, Mutation을 실행하기 위해 client로부터 전달받은 데이터가 type에 정의된 필드의 형(Data type)과 다를 경우 GraphQL에서는 error를 리턴하기 때문에 동적 타입 언어에서도 형(Data type)에 대한 안정성을 확보할 수 있다.
